# 会员订单支付页体验补齐设计

日期：2026-04-19

状态：

- draft

关联文档：

- [order-payment-ui-api-contract.md](./order-payment-ui-api-contract.md)

## 1. 目标

补齐会员端订单详情页中的支付区体验，使用户在“待支付”状态下能够：

- 看到当前订单应付金额
- 只看到后台当前实际允许的支付方式
- 在支付回跳后得到明确的结果提示

本次改动保持“下单页创建订单，订单详情页发起支付”的既有流程，不新建独立支付页。

## 2. 当前缺口

当前会员端支付区存在以下问题：

- 前端写死展示 `alipay` 和 `wxpay`，与后台实际支付方式配置可能不一致
- 订单详情页待支付状态下不展示应付金额，用户从订单中心返回继续支付时缺少关键确认信息
- 支付成功或取消后都回到同一订单页，但页面没有明确的成功/取消提示

## 3. 已锁定决策

- 订单支付语义仍然是“为单次检测服务订单付款”，不是充值、订阅或额度购买
- 发起支付接口仍使用 `POST /api/orders/:id/pay`
- 支付成功后的业务语义不变：
  - `allergy_order.payment_status = paid`
  - `allergy_order.order_status = paid`
  - 写入时间线事件 `payment_completed`
- 前端不暴露 `topup`、`quota`、`token` 等平台语义

## 4. 非目标

- 不新增新的支付渠道
- 不改动 `epay` webhook 验签或订单支付完成语义
- 不新做独立收银台页
- 不处理退款、售后、对账

## 5. 页面与状态变更

### 5.1 下单页

- 维持当前“创建订单”职责不变
- 文案继续说明用户会在订单详情页继续完成支付

### 5.2 订单详情页

在订单 `payment_status = pending` 时：

- 支付区显示订单应付金额
- 支付方式列表改为后端返回的动态列表
- 如果后端返回空列表，禁用支付动作并提示管理员尚未配置可用支付方式
- 用户发起支付后继续沿用后端返回的跳转表单/地址

在支付回跳后：

- 支持识别 `success` / `cancelled` 两类结果
- 成功时展示明确成功提示，随后以订单状态角标和支付时间作为最终事实来源
- 取消时展示“支付未完成，可重新选择方式继续支付”

在订单 `payment_status != pending` 时：

- 不再展示支付方式选择，仅展示支付完成信息

## 6. 数据与接口影响

本次需要补充会员端订单读接口返回的数据，而不新增独立支付配置接口：

- `GET /api/orders/:id`
  - 补充订单价格快照字段
  - 补充币种字段
  - 补充当前可用支付方式列表

支付请求接口保持不变：

- `POST /api/orders/:id/pay`

## 7. 风险点

- 后台支付方式配置为空时，订单详情页不能继续假设至少存在支付宝或微信支付
- 支付回跳提示只能作为用户反馈，最终支付结果仍必须以 `/api/orders/:id/pay-status` 返回值为准
- 历史订单价格必须继续使用订单快照，不应回读当前商品目录价格

## 8. 验收标准

- 待支付订单详情页能展示订单应付金额
- 待支付订单详情页的支付方式来源于后端返回，而不是前端写死
- 当后台无可用支付方式时，页面给出明确提示且不会提交无效支付请求
- 支付成功回跳后，页面能展示成功提示，并显示已支付状态
- 支付取消回跳后，页面能展示取消提示，且订单仍保持待支付
