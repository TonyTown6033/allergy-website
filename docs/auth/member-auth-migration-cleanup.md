# `new-api` + `Allergy` 会员账号迁移与旧会员清理说明

日期：2026-04-15

关联文档：

- [member-auth-design.md](./member-auth-design.md)
- [member-auth-api-contract.md](./member-auth-api-contract.md)

## 1. 范围

这份文档只描述账号体系切换时需要的数据迁移、切换顺序和旧会员清理方案。

目标：

- 支撑公共站从“验证码登录”切换到“用户名/邮箱 + 密码”
- 处理旧普通会员账号与新注册体系的冲突
- 保证管理员账号不受影响

不包含：

- 业务代码实现细节
- 履约订单迁移
- 历史报告迁移
- 退款与财务数据处理

## 2. 背景

旧体系存在以下特征：

- 用户可通过邮箱验证码直接登录
- 首次登录会自动创建账号
- 没有独立注册流程
- 没有稳定的邮箱验证完成语义
- 没有完整密码登录闭环

新体系的关键约束变成：

- 注册前必须完成邮箱验证
- 邮箱必须唯一
- 用户名必须唯一
- 日常登录必须使用密码
- 只有有效 `member_profile` 的会员可登录公共站

因此，旧普通会员账号如果继续保留，会在新体系下带来几个直接问题：

- 邮箱唯一冲突
- 用户可能没有密码
- 历史账号不满足新准入规则
- 前端无法判断该提示注册、登录还是找回密码

## 3. 已确认的业务决策

以下业务决策已确认：

- 旧普通会员账号不做平滑迁移
- 旧普通会员账号通过脚本清理
- 不迁移旧订单 / 报告 / 历史会员资料
- 上线后用户按新体系重新注册
- 管理员账号不能被清理
- 公共站切换完成后移除验证码登录主流程

这意味着当前方案不是“增量升级老会员”，而是“清理旧会员后进入新体系”。

## 4. 涉及的数据对象

### 4.1 保留对象

- 后台管理员 `user`
- 管理员相关权限与配置
- 履约订单数据
- 报告数据
- 管理端业务数据

### 4.2 清理对象

建议清理以下与旧普通会员登录直接相关的数据：

- 旧普通会员 `user`
- 对应 `member_profile`
- 对应 `member_session`
- 对应 `email_login_code_store`

### 4.3 谨慎项

若线上已经存在与旧普通会员关联的订单、报告或附件，当前方案是“不迁移”，但执行前仍需确认：

- 删除会员账号是否会破坏外键约束
- 是否需要先解除关联或保留壳账号

如果存在强外键依赖，应先调整为安全删除方案，而不是直接硬删。

## 5. 推荐切换策略

建议采用“先支持新体系，再切流，再清旧数据”的顺序。

### 5.1 阶段一：后端支持新体系

先完成但不立即开放给用户：

- 新注册接口
- 新密码登录接口
- 新找回密码接口
- 新准入校验
- `member_profile.email_verified_at`

### 5.2 阶段二：前端准备新页面

准备公共站新页面：

- `/login`
- `/register`
- `/forgot-password`

并完成：

- `AuthContext` 切换
- 原验证码登录入口下线

### 5.3 阶段三：执行旧会员清理

在正式切流前执行清理脚本：

1. 先 `dry-run`
2. 人工确认待删账号集合
3. 正式删除
4. 核对管理员账号仍可用

### 5.4 阶段四：正式切流

切换公共站入口为新登录体系：

- 停用旧 `send-code` 登录流程
- 启用注册 / 登录 / 找回密码页面

### 5.5 阶段五：观察期

上线后重点观察：

- 注册成功率
- 登录失败原因分布
- 找回密码成功率
- 是否有管理员被误拦截

## 6. 数据库变更建议

### 6.1 `member_profile`

建议新增字段：

- `email_verified_at`

目标：

- 明确记录邮箱完成验证的时间
- 登录准入判断不再靠隐式语义

### 6.2 `email_login_code_store`

无需改表结构也可先落地，但要调整用途约定：

- `register_verify`
- `password_reset`

建议停止新增：

- `login`

### 6.3 `user`

需要确保业务上满足：

- 邮箱唯一
- 用户名唯一

如果数据库当前未完全保证，则至少要在应用层强校验。

## 7. 旧会员识别策略

清理脚本需要先判定哪些账号属于“可删除的旧普通会员”。

建议采用保守识别，不要只看单个字段。

### 7.1 推荐识别条件

建议至少同时参考：

- 非管理员角色
- 存在 `member_profile`
- 属于公共站会员范围
- 来源于旧验证码登录时期

### 7.2 强保护条件

以下任一条件命中时，不应删除：

- 管理员角色
- 明确配置为内部运营账号
- 明确标记为测试管理员账号

### 7.3 不建议的粗暴做法

不建议直接按“有 `member_profile` 就删”，因为这可能误伤未来已注册的新会员，或误删用于联调的内部账号。

因此脚本应支持明确筛选条件和结果预览。

## 8. 清理脚本设计

建议提供独立脚本或命令，例如：

- `allergy-member-cleanup --dry-run`
- `allergy-member-cleanup --execute`

当前仓库已落地脚本：

- `go run ./bin/allergy_member_cleanup.go`
- `go run ./bin/allergy_member_cleanup.go --execute`

当前实现默认采用保守筛选：

- 仅处理 `member_profile.email_verified_at IS NULL` 的旧普通会员
- 默认跳过管理员账号
- 默认跳过已经关联 `allergy_order` 的账号

### 8.1 必备能力

- `dry-run`
- 输出待删账号数量
- 输出待删邮箱 / 用户名样例
- 输出受影响的 `member_profile / member_session / code` 数量
- 正式执行删除
- 执行结果摘要

### 8.2 推荐输出结构

`dry-run` 建议输出：

- 待删会员数量
- 待删会话数量
- 待删验证码记录数量
- 样例邮箱前若干条
- 被保护跳过的管理员数量

### 8.3 删除顺序建议

若采用硬删除，推荐顺序：

1. `member_session`
2. `email_login_code_store`
3. `member_profile`
4. `user`

前提：

- 确认没有更深层外键依赖

若存在外键依赖，则应改为：

- 先解除依赖
- 或保留 `user` 壳记录，仅清理公共站登录能力

### 8.4 幂等要求

脚本应尽量幂等：

- 已删除对象再次执行不报致命错误
- `dry-run` 和正式执行基于同一筛选逻辑

## 9. 风险点

### 9.1 订单 / 报告外键风险

如果订单或报告直接引用 `user_id`，硬删 `user` 可能失败，或导致历史数据不可访问。

应在正式执行前先确认：

- `allergy_order` 是否直接依赖 `user`
- `lab_report` 是否间接依赖 `user`

### 9.2 管理员误删风险

这是最高优先级风险。

因此脚本必须：

- 明确区分管理员和普通会员
- 在 `dry-run` 输出保护对象数量
- 在正式执行前提供二次确认机制

### 9.3 切流窗口风险

如果先清掉旧会员，再部署新登录体系，用户会直接无法进入站点。

因此不能采用：

- 先删账号
- 后开发新体系

正确顺序必须是：

- 先具备新体系
- 再清旧账号
- 再切流

### 9.4 测试环境与生产环境差异

如果测试环境管理员账号结构与生产不同，脚本在测试环境通过，不代表线上绝对安全。

正式上线前应至少确认：

- 线上管理员角色标识
- 内部测试账号标识
- 是否存在共享邮箱或混合用途账号

## 10. 回滚思路

当前方案本质是破坏性切换，严格意义上的“数据回滚”成本较高。

因此建议把回滚思路建立在“切流可回退”，而不是“删后恢复”。

### 10.1 推荐回滚策略

- 新前端入口可回退到旧页面
- 新后端认证路由可临时恢复旧逻辑
- 清理脚本必须先做 `dry-run`

### 10.2 不应依赖的回滚方式

不应假设删除后的普通会员数据可以轻松恢复，除非你明确具备：

- 数据库备份
- 恢复演练
- 可接受的数据恢复窗口

## 11. 最低执行清单

正式上线前建议按这个顺序核对：

1. 后端新认证接口已完成并通过测试
2. 前端新登录 / 注册 / 找回密码页面已完成联调
3. 管理员公共站拦截逻辑已验证
4. `member_profile.email_verified_at` 已接入
5. 清理脚本 `dry-run` 结果已人工确认
6. 已确认不会误删管理员
7. 已确认删除不会破坏订单 / 报告外键
8. 已准备数据库备份或替代保护措施
9. 正式执行清理
10. 切换公共站入口
11. 上线后观察注册 / 登录 / 找回密码链路

## 12. TDD 与验证建议

建议至少覆盖以下测试：

- 识别脚本不会把管理员纳入待删集合
- `dry-run` 不会产生删除
- 正式执行只清普通会员
- 清理后旧会员邮箱可以重新注册
- 清理后管理员仍能正常进入后台
- 切换后旧验证码登录入口不可用
- 新注册用户可以完成登录与下单

## 13. 非目标

这份迁移说明不覆盖：

- 历史订单重新归属到新账号
- 历史 PDF 报告重新绑定
- 自动通知老会员数据已失效
- 批量导入外部用户

如果未来要做“老会员资产迁移”，那将是另一套独立方案，不应混进当前这一轮账号体系切换。
