# `new-api` + `Allergy` 集成设计

日期：2026-04-13

> 状态说明
> - 这份文档保留第一阶段兼容接入设计
> - 其中认证部分基于早期“邮箱验证码登录”假设
> - 当前会员账号体系请以 [../auth/member-auth-design.md](../auth/member-auth-design.md) 为准

## 1. 目标与约束

本设计的目标是：

- 以 `new-api` 作为后端平台底座
- 保留 `Allergy` 当前面向用户的前端形态和页面心智
- 先做架构设计，不在本轮修改业务代码
- 产出后续可以直接指导实施的落地文档

本轮明确约束：

- 不改 `Allergy` 现有前端代码
- 不改 `new-api` 现有代码
- 只讨论集成方案、边界和实施顺序

### 1.1 本轮已确认的方向

- 接受三域模型，但优先采用最简单的部署形态
- 登录方式改为邮箱验证码，不再把手机号作为主登录方式
- `Allergy` 的核心不是 token / quota 业务，而是检测订单、采样盒、送检、报告展示
- 第一阶段内容继续使用简单 JSON 承接
- 图片方案先保持简单，不强行上对象存储后台

## 2. 当前现状

### 2.1 `Allergy` 现状

`Allergy` 本质上是一个轻量业务站点，前端为 React + Vite，后端为小型 Express 服务。

当前前端依赖的接口非常少，主要是：

- `GET /api/hero`
- `GET /api/testimonials`
- `GET /api/articles`
- `GET /api/products`
- `POST /api/auth/send-code`
- `POST /api/auth/login`
- `GET /api/auth/me`

当前认证模型是：

- 前端本地保存 `token` 和 `phone`
- 登录响应格式为 `{ success, token, phone }`
- 后续通过 `Authorization: Bearer <token>` 调用 `/api/auth/me`

当前内容模型是：

- 首页主图
- 用户故事
- 文章列表
- 产品列表
- 图片路径依赖 `/images/*` 或外链

这说明 `Allergy` 当前是“展示站点 + 轻认证 + 少量内容接口”的模式。

### 2.2 `new-api` 现状

`new-api` 是完整的平台型系统，后端为 Go + Gin，内置前端控制台、用户系统、支付、订阅、令牌、日志、模型路由、OAuth、2FA、Passkey 等能力。

已经具备可复用的底座能力：

- 用户体系
- 会话管理
- OAuth / 2FA / Passkey
- 支付与订阅
- 额度与分组
- 日志与统计
- 后台控制台
- 配置项存储 `Option`
- 一定程度的页面内容配置接口，例如：
  - `/api/home_page_content`
  - `/api/notice`
  - `/api/about`
  - `/api/pricing`

同时，`new-api` 还支持通过 `FRONTEND_BASE_URL` 将非 API 路由重定向到外部前端，这一点对“保留 `Allergy` 前端”很关键。

## 3. 两个项目的核心差异

如果直接把 `Allergy` 前端接到 `new-api`，会立刻遇到以下不匹配。

### 3.1 认证模型不匹配

`Allergy` 现在用的是：

- 手机号 + 验证码
- 返回业务 token
- Bearer Token 调 `/api/auth/me`

`new-api` 原生提供的是：

- 用户名/密码登录
- OAuth 登录
- 基于 Session/Cookie 的 Web 登录流程
- `/api/user/self` 这类控制台接口

这两种模型不是简单改个接口路径就能兼容。

### 3.2 内容模型不匹配

`Allergy` 需要的是品牌站内容：

- Hero
- Testimonials
- Articles
- Products

`new-api` 当前没有现成的“过敏品牌站内容域模型”，只有通用配置项和控制台内容配置能力。

### 3.3 部署模型不匹配

`Allergy` 当前是：

- 一个独立前端
- 通过 `VITE_API_URL` 指向后端

`new-api` 当前更偏向：

- 自带控制台 Web
- 同服务承载 API 和 Web

如果不做边界设计，公共站点和后台控制台会互相污染路由与部署策略。

### 3.4 静态资源管理不匹配

`Allergy` 当前图片来自：

- 本地静态目录 `/images/*`
- 外部 URL

`new-api` 没有现成的品牌站图片资源管理流水线。它更适合读取外部 URL 或配置值，不适合继续沿用 Express 本地静态目录思路。

## 4. 设计原则

为了降低集成风险，建议遵守以下原则。

### 原则一：前端尽量少动，后端优先兼容

既然当前目标不是重做前端，后端就不应该强迫前端马上迁移到 `new-api` 原生控制台 API。

第一阶段应该让 `new-api` 提供一层“兼容接口”，让前端只做必要的低改动，例如把手机号登录切到邮箱登录。

### 原则二：复用 `new-api` 的平台能力，不复用它的控制台交互模型

应该复用：

- 用户、订阅、支付、日志、AI 网关、配置能力

不应该直接复用：

- 控制台前端的登录流程
- 控制台页面结构
- 面向管理员的接口语义

换句话说，复用平台，不复用 Console UX。

### 原则三：公共站点、API、后台控制台物理分离

建议至少逻辑上拆成三块：

- 公共站点前端
- 面向站点的业务 API
- 后台运营控制台

这能避免将来在路由、鉴权、缓存、SEO、管理员入口上互相牵扯。

### 原则四：先小步兼容，后统一模型

第一阶段不要追求“一次性改成 `new-api` 原生方式”。

先兼容上线，再逐步统一：

- 先兼容 `Allergy` 的现有 API 形态
- 再逐步把内容管理、会员、AI 功能沉入 `new-api` 的统一域模型

## 5. 推荐目标架构

### 5.1 推荐部署拓扑

建议采用三域或三入口模型：

- `www.allergy.xxx`
  - 部署 `Allergy` 前端静态站点
- `api.allergy.xxx`
  - 部署 `new-api` 后端服务
  - 对外提供 `Allergy` 业务 API 和 AI 网关能力
- `console.allergy.xxx`
  - 部署 `new-api` 控制台
  - 供运营、配置、内容管理、支付管理使用

这样做的好处：

- 公共站点和后台控制台边界清晰
- 不需要把 `new-api` 的 Web 路由强行和品牌站揉在一起
- 后续可以单独给公共站做 CDN、SEO、缓存策略

### 5.1.1 最简单的落地方案

三域不等于三台机器。最简单的形态可以是：

- 一台服务器，或一套 `docker compose`
- 一个反向代理，例如 `Nginx` 或 `Caddy`
- 一个 `new-api` 进程
- 一个 `Allergy` 静态站点

路由方式建议如下：

- `www.allergy.xxx` 指向 `Allergy` 打包后的静态文件
- `api.allergy.xxx` 只转发 `/api/*` 到 `new-api`
- `console.allergy.xxx` 整站转发到 `new-api` 控制台

这样虽然是三域，但运行单元其实只有两个：

- `Allergy` 前端静态站点
- `new-api` 后端与控制台

这是我建议的最简方案。它保留了边界，又不会把部署复杂度抬得太高。

### 5.2 不推荐的拓扑

不建议第一阶段就做“同域同进程混合承载”：

- `/` 走 `Allergy`
- `/api` 走 `new-api`
- `/console` 再塞回 `new-api` 控制台

原因不是做不到，而是：

- `new-api` 自带 Web 路由和 `NoRoute` 行为
- `FRONTEND_BASE_URL` 会影响非 API 路由
- 后续调试、缓存、重定向、Cookie 策略会更复杂

如果未来一定要单域，也建议先在架构上完成域模型统一，再做入口合并。

## 6. 推荐的集成方式

### 6.1 结论

推荐采用：

- `Allergy` 前端保持独立
- `new-api` 作为业务 API 和平台底座
- 在 `new-api` 内增加一层 `Allergy Compatibility API`

也就是：

前端不直接适配 `new-api` 控制台接口，而是让 `new-api` 提供一组面向 `Allergy` 业务的接口。

### 6.2 为什么推荐兼容层

因为这是当前约束下成本最低、风险最小的方案。

它满足三件事：

- 公共站点可以尽量少改
- 后端逐步迁移到 `new-api`
- 后面还能继续演进到统一会员、订单、报告和 AI 业务模型

## 7. 接口兼容层设计

建议在 `new-api` 中新增一个 Allergy 业务域，先暴露与当前前端尽量贴近的接口。

### 7.1 Phase 1 兼容接口

| 当前接口 | Phase 1 在 `new-api` 中的角色 | 后续演进方向 |
|---|---|---|
| `GET /api/hero` | 从配置型内容读取首页主图 | 后续可并入品牌站内容模型 |
| `GET /api/testimonials` | 从配置型内容或内容表读取故事列表 | 后续升级为可排序、可上下线内容 |
| `GET /api/articles` | 从配置型内容或文章表读取摘要列表 | 后续升级为完整文章 CMS |
| `GET /api/products` | 从配置型内容或产品表读取商品卡片 | 后续升级为商品/套餐域模型 |
| `POST /api/auth/send-code` | 发送邮箱登录验证码 | 后续可升级为 magic link 或双重验证 |
| `POST /api/auth/login` | 验证邮箱验证码并建立会员登录态 | 后续可统一到会员中心 |
| `GET /api/auth/me` | 获取当前会员信息 | 后续与会员档案打通 |

### 7.2 检测业务的下一批接口

由于真实业务是检测闭环，不是 token 业务，后续应该增加的核心接口是：

- `POST /api/orders`
- `GET /api/orders`
- `GET /api/orders/:id`
- `GET /api/reports`
- `GET /api/reports/:id`
- `GET /api/me/timeline`

这批接口的职责不是暴露平台能力，而是描述真实业务状态：

- 订单是否已支付
- 采样盒是否已寄出
- 样本是否已回寄
- 检测机构是否已出报告
- 用户是否可以查看报告与解释

### 7.3 兼容层的边界

兼容层只解决“当前前端能无感切到 `new-api`”。

兼容层不应直接暴露：

- `new-api` 的 Console 用户管理接口
- `token` 管理接口
- 原始模型路由和渠道管理接口
- 面向管理员的订阅/支付后台接口

公共站点应该拿到面向站点的业务接口，而不是平台原始接口。

## 8. 认证方案设计

这里是整个集成中最重要的一块。

### 8.1 先澄清这里的 token 含义

这里必须先去掉一个歧义。

前面提到的 `token`，指的是“登录态标识”或“session token”，不是：

- AI 调用额度
- API key
- `new-api` 的渠道 token
- 面向用户售卖的配额 token

`Allergy` 的核心业务仍然是：

- 用户购买检测盒
- 我们寄送采样盒
- 用户完成采样并寄回
- 检测机构出具报告
- 用户查看报告

所以登录只是辅助能力，不是产品本身。

### 8.2 推荐的 Phase 1 登录方案

推荐采用“邮箱验证码 + 登录态 token”模式：

- 用户输入邮箱
- `POST /api/auth/send-code` 发送邮箱验证码
- `POST /api/auth/login` 使用 `email + code` 登录
- 后端返回一个登录态标识，名称可以继续叫 `token`，也可以在实现时改名为 `session_token`
- 这个值只表示“当前会员已登录”，不表示任何 AI 配额或商品额度

之所以仍建议保留这种登录态，是因为对于跨域 SPA 来说，它比一上来切复杂 Cookie Session 更直接。

### 8.3 会员身份如何落到 `new-api`

推荐做法：

- 以 `new-api` 的 `user` 体系作为账号主表
- 当前决策是：会员直接挂到这张主表上，不再额外引入独立的业务会员主键
- 以邮箱作为主登录标识
- 新增 `member_profile` 或类似附表保存业务字段
- `username` 作为内部字段使用，不把它暴露成业务主标识

不推荐的做法：

- 直接把邮箱本地部分硬塞成核心业务主键
- 直接复用 `new-api` 的 `access_token` 作为站点会员登录态

原因：

- `username` 是平台账号概念，不适合作为长期业务主键
- `access_token` 在 `new-api` 里更偏管理/API 令牌语义，和站点会员登录态不是一回事

### 8.4 推荐的会员认证落地形态

第一阶段建议在设计上预留以下能力：

- `member_profile`
  - `user_id`
  - `email`
  - `phone`
  - `nickname`
  - `avatar`
  - `status`
- `email_login_code_store`
  - `email`
  - 验证码
  - 过期时间
  - 使用状态
- `member_session`
  - 登录态 token 或 JWT
  - `user_id`
  - 过期时间
  - 设备信息

这样做的好处是：

- 登录形态简单
- 会员主键简单，后台和权限体系更容易复用
- 后续仍可逐步收敛到统一会员域

同时要补一条边界规则：

- 公共站会员的准入标记是存在可用的 `member_profile`
- 邮箱验证码登录只面向公共站会员
- 后台管理员账号如果没有可用 `member_profile`，则不能走公共站登录
- 如果管理员需要测试前台流程，使用独立会员邮箱，不与后台邮箱共用

## 9. 内容模型设计

### 9.1 Phase 1 建议：先用配置型内容承接

因为当前 `Allergy` 的内容量很小，第一阶段没有必要立即上完整 CMS。

建议先用 `new-api` 的配置能力承接以下内容块：

- `allergy.hero`
- `allergy.testimonials`
- `allergy.articles`
- `allergy.products`

每个配置项保存 JSON 数据，接口层负责转换成当前前端需要的响应格式。

这种做法适合：

- 快速接管原有 Express 后端
- 先完成迁移
- 先把品牌站内容纳入统一后台

### 9.2 Phase 2 再升级为内容表

当出现以下需求时，再升级为独立表模型：

- 文章越来越多
- 需要草稿/发布/下线
- 需要排序、标签、搜索、分类
- 需要内容审核或多语言
- 需要运营同学可视化管理

届时再拆分为：

- `allergy_articles`
- `allergy_products`
- `allergy_testimonials`
- `allergy_home_blocks`

### 9.3 图片方案先保持简单

你刚才说“不明白对象存储/CDN”，这个可以简单理解：

- 对象存储：就是云上的图片文件夹，例如 OSS、S3、COS
- CDN：就是把图片缓存到离用户更近的节点，让访问更快

但第一阶段其实不用强上这套。

第一阶段最简单的做法是：

- JSON 里的图片字段先存固定 URL
- 这些 URL 可以先指向前端静态资源，或者指向固定的 HTTPS 图片地址
- 后台先不做图片上传能力

什么时候才需要对象存储：

- 运营要频繁换图
- 需要后台上传图片
- 图片越来越多，不想每次换图都重新发版

所以这里的结论不是“必须上对象存储”，而是：

- Phase 1 不需要
- Phase 2 如果开始做后台上传，再考虑

## 10. 检测业务与平台复用建议

### 10.1 核心业务主流程

当前真实业务应该按这个流程建模：

- 用户浏览站点并购买检测服务
- 系统创建订单
- 运营寄送采样盒
- 用户完成采样并寄回
- 后台登记“样本已回寄”
- 检测机构接收样本并出具报告
- 运营导入或确认报告
- 用户登录后查看报告
- 后续可叠加专家解读或 AI 解读

这里最核心的是“订单与报告状态流转”，不是 token 消耗。

### 10.2 推荐的核心领域模型

建议后续逐步补齐这些表或等价模型：

- `member_profile`
  - 会员档案
- `allergy_order`
  - 订单信息、支付状态、收货信息
- `sample_kit`
  - 采样盒编码、寄送状态、回寄状态
- `lab_submission`
  - 送检批次、检测机构、接收时间
- `lab_report`
  - 报告摘要、报告状态、报告发布时间
- `report_asset`
  - PDF、图片、结构化结果 JSON
- `report_timeline`
  - 用于给用户展示订单到报告的全过程

### 10.3 哪些能力适合直接复用 `new-api`

建议直接复用：

- `user` 体系
- 登录安全能力
- 支付渠道接入能力
- 支付 webhook 验签与回调能力
- 配置项存储能力
- 管理后台能力
- 审计和日志能力

但支付这块要明确边界：

- 复用 `new-api` 的支付接入层
- 不直接复用 `TopUp` / `Recharge` / quota 增减语义
- 支付成功后更新 `allergy_order.payment_status = paid`
- 后续履约仍走人工流程，例如寄送采样盒、上传报告、手工补发邮件

### 10.4 哪些概念不要直接照搬到业务层

不建议把以下平台概念直接暴露给 Allergy 用户：

- token quota
- model quota
- channel
- relay
- model pricing

这些是平台内部概念，不是检测业务概念。

### 10.5 AI 在这个业务里的正确位置

这里的 `Allergy.AI`，指的是一个未来可选的增值能力，不是当前必须建设的功能。

它的意思是：

- 当用户拿到检测报告之后
- 系统可以额外提供报告解读、饮食建议、过敏知识问答

如果后面要接 `Allergy.AI`，建议把它放在“报告解读”和“问答辅助”层，而不是主交易流本身。

也就是：

- 先把订单、采样、报告闭环跑通
- 再让 AI 参与报告说明、饮食建议、过敏知识问答

这样 AI 是增值能力，而不是核心交易依赖。

### 10.6 后台应该承担什么

推荐让 `new-api` 控制台逐步承担这些后台职责：

- 首页内容配置
- 产品与套餐配置
- 订单状态管理
- 采样盒寄送登记
- 报告上传与发布
- 会员查看权限控制
- 后续 AI 功能开关与权限配置

## 11. 推荐的分阶段落地顺序

### Phase 0：设计冻结

先明确以下决定：

- 域名拓扑采用三域模式，但按最简部署实现
- 登录方式改为邮箱验证码
- 会员直接与 `new-api user` 主表绑定
- 图片第一阶段不做对象存储后台
- 商品先按单次检测订单建模，而不是 token / quota 建模
- 首版采用线上支付 + 人工履约
- 支付复用 `new-api` 支付接入层，不复用 `TopUp/Recharge` 业务语义
- 首发支付渠道必须包含易支付

这一步不写业务代码，只冻结设计选择。

### Phase 1：低改动迁移

目标：

- 用 `new-api` 接管品牌站后端
- 先完成站点内容和会员登录
- 先完成检测服务线上售卖
- 为后续订单与报告闭环打基础

实施重点：

- 在 `new-api` 中实现兼容接口
- 内容先走配置型 JSON
- 认证改为邮箱验证码 + 登录态 token
- 前端只通过 `VITE_API_URL` 指向 `new-api`
- 图片先用固定 URL 或前端静态资源
- 接入线上支付，首发启用易支付
- 支付成功后仅更新检测订单支付状态，不做 quota 充值
- 订单与报告模型先完成最小结构设计

交付结果：

- `Allergy` 站点主体 UI 基本不变
- 原 Express 后端可退出
- 账号、内容、配置开始沉入 `new-api`
- 用户可以直接线上支付购买检测服务
- 后续检测闭环有了明确承载位置

### Phase 2：订单与报告闭环

目标：

- 让真实业务跑起来
- 让用户可以查看自己的检测进度和报告

实施重点：

- 建立订单、采样盒、送检、报告表
- 增加后台订单与报告管理能力
- 第一版先支持后台上传 PDF 报告
- 支持后台手工补发 PDF 报告到用户邮箱
- 支持用户查看个人检测进度时间线
- 报告页面第一版支持 PDF 预览和下载
- 用户回寄样本先由后台手工登记，不要求前台自助提交
- PDF 预览和下载走受保护接口，由前端带登录态拉取 PDF `Blob`
- 若订单存在多份报告，只向用户暴露当前有效版本

### Phase 3：内容与会员后台化

目标：

- 让运营可更稳定地维护站点与会员资料

实施重点：

- 内容从 Option JSON 逐步升级为专用表
- 增加后台内容编辑能力
- 增加会员档案、地址、通知偏好等资料管理

### Phase 4：可选的 AI 增值能力

目标：

- 在主业务稳定后，再考虑是否接入 `Allergy.AI`

实施重点：

- 接入 `new-api` 的模型网关能力
- 基于会员等级或服务套餐控制 AI 权限
- 记录咨询日志、报告解读记录、推荐结果

## 12. 关键风险

### 风险一：把检测业务误建模成 token / quota 业务

如果直接沿用 `new-api` 的平台计费心智去建模 Allergy 业务，会忽略真正重要的状态：

- 订单
- 物流
- 样本
- 检测机构
- 报告

这会导致底座虽然强，但业务模型一开始就歪掉。

### 风险二：过早强推 `new-api` 原生登录流程

如果直接要求 `Allergy` 前端改用 `new-api` 的 Session/Cookie 登录，会同时引入：

- 前端重构
- 跨域 Cookie 配置
- 安全策略调整
- 登录态迁移

这会把简单集成变成全面改造。

### 风险三：把品牌站直接绑定到平台原始 API

如果前端直接调用 `new-api` 原生用户、支付、模型接口，短期看省事，长期会造成：

- 接口语义不稳定
- 前端暴露平台概念
- 平台升级牵动品牌站

所以必须有业务层。

### 风险四：长期滥用 Option JSON 做 CMS

Option JSON 很适合第一阶段，但不适合一直承担完整内容系统。

一旦出现复杂运营需求，应尽快升级到专用内容表。

### 风险五：过早建设复杂图片系统

如果第一阶段就上对象存储、后台上传、图片审核、CDN 回源，会把不必要的复杂度提前。

当前阶段更合理的做法是：

- 先用固定图片 URL
- 业务跑通后再决定是否建设完整图片系统

## 13. 当前建议的最终结论

当前最合理的路线不是“让 `Allergy` 前端直接吃 `new-api` 原生 Console API”，而是：

- 保留 `Allergy` 前端
- 用 `new-api` 承接后端平台能力
- 在 `new-api` 中新增一层 Allergy 业务兼容 API
- 第一阶段先接管内容与登录
- 内容先用配置型 JSON 承接
- 认证改为邮箱验证码 + 登录态 token
- 业务核心围绕订单、采样盒、送检、报告来建模
- 后续再叠加会员、后台和 AI 解读能力

这条路线最符合当前约束，也最容易分阶段落地。

## 14. 建议立即确认的决策

当前已确认的决策：

1. 接受三域模型，但走最简单部署。
2. 登录改为邮箱验证码。
3. 业务核心是检测订单与报告闭环，不是 token / quota 业务。
4. 第一阶段内容先放 `Option JSON`。
5. 第一阶段图片先保持简单，不上复杂对象存储方案。
6. 商品第一版按“单次购买检测服务”建模。
7. 报告第一版先只上传 PDF，并最好支持直接发送到用户邮箱。
8. 用户端第一版最好展示检测进度时间线。
9. `Allergy.AI` 目前不是当前主线，后续再决定是否建设。
10. 会员直接挂在 `new-api user` 主表。
11. PDF 报告支持后台手工补发到用户邮箱。
12. 报告页面第一版支持 PDF 预览和下载。
13. 首版采用线上支付，付款后人工履约。
14. 支付复用 `new-api` 现有支付接入层，不复用 `TopUp/Recharge` 的充值语义。
15. 首发支付渠道必须包含易支付。
16. PDF 预览和下载不走公开直链，走带会员登录态的受保护接口。
17. 报告补发只允许发到账号邮箱或订单通知邮箱，不允许任意输入邮箱。
18. 公共站会员登录与后台管理员登录严格分流，以 `member_profile` 作为公共会员准入标记。
19. 用户回寄样本第一版由后台登记，时间线需要体现“已回寄”。
20. 一个订单可有多个报告版本，但同一时间只能有一份当前有效报告。

当前还没必要继续细化的点，后面实施时再定：

1. 邮件发送是接现成 SMTP，还是第三方事务邮件服务。
2. 检测进度时间线需要展示到多细，例如是否区分“已寄出采样盒”和“检测机构已签收样本”。
3. 报告页后续是否增加几个简单摘要字段，作为 PDF 之外的结构化补充。
